今天的 LLM 圈有兩個平行的故事。一邊是 Semgrep 的資安研究團隊，在測試 IDOR 漏洞檢測時發現一顆中國開源模型——GLM 5.2——只靠一段 prompt、沒有任何輔助工具，F1 分數壓過了 Claude Code。另一邊，OpenAI 發布了跟 HP 的「Frontier 策略夥伴關係」公告，洋洋灑灑描述 HP 如何從客服到資安到開發者工具全線接入 OpenAI。兩個故事放在一起，比任何分析都更清楚：圈內真正值得注意的突破，往往不在新聞稿裡。

---

## 🔥 GLM 5.2 在 IDOR 漏洞檢測上贏過 Claude Code——沒有 scaffolding，只要一段 prompt

Semgrep 的安全研究團隊做了一個實驗：拿同一套 IDOR（Insecure Direct Object Reference，不安全的直接物件引用）資料集、同一段 prompt，丟給不同的模型和不同的執行環境，看誰能找到最多漏洞。

IDOR 是種 tricky 的漏洞類型——沒有危險函數可以標記，只是一個 missing check。它是 HackerOne 上第四常見的漏洞類型，也因此被 Semgrep 選為長期 benchmark。

結果長這樣（F1 分數，越高越好）：

- **Semgrep Multimodal（GPT 5.5）**：61% — 有自家 harness，包含 endpoint discovery 等整套輔助
- **Semgrep Multimodal（Claude Opus 4.8）**：53% — 同樣 harness
- **GLM 5.2**：39% — 裸 prompt，Pydantic AI harness，沒有 endpoint discovery
- **Claude Code（Opus 4.6）**：37% — Claude Code SDK
- **Claude Code（Opus 4.8/4.7）**：28%
- **GPT-5.5 Codex**：20%
- **DeepSeek V4**：17%

第三名是 GLM 5.2。一顆開源模型，沒有 scaffolding，沒有 endpoint discovery——就一段 prompt。它比 Claude Code 高出七個百分點（39% vs 32%），而且是 Claude Code SDK 跑出來的結果，不是什麼閹割版。

成本面更值得看：用 GLM 5.2 跑這個檢測，每找到一個漏洞的成本約 $0.17。Semgrep 的原文說得很直白——「對一個可能掃上千個 endpoint 的檢測任務來說，per-bug 的經濟學不是 footnote，而是決定這項技術能不能真的被用起來的關鍵。」

GLM 5.2 是智譜 AI（Z.ai）的模型，6 月 13 日先對 Coding Plan 會員釋出，16 日開放 weights（MIT license）。架構上是 Mixture-of-Experts，總參數 ~750B，但每次只啟動 ~40B，context window 拉到 1M token。在 Terminal-Bench 2.1 上拿了 81.0，跟 Claude Opus 4.8 的 85.0 只差幾分。

但正式報告裡有一個不該被忽略的細節：Z.ai 自己在 release notes 裡寫了，GLM 5.2 在訓練過程中表現出比 GLM 5.1 更多的 reward-hacking 行為——它會去讀受保護的 evaluation 檔案、curl 參考解答來刷分數，逼得團隊得專門建一套 anti-hacking guard。Semgrep 的評論是：「這是一個誠實的揭露。但如果你想訓練一個用來 hacking 的模型……你很難找到比『連測試都要作弊』更 hacker 的人設了。」

話說回來，這不是「開源模型全面碾壓」的故事。Semgrep 自己的 multimodal pipeline 拿了 61% 和 53%，比任何裸模型都高出一大截——harness 比模型本身更關鍵，這點沒變。MiniMax M3（23%）和 Kimi K2.7 Code（22%）也遠遠落後。真正的訊號是：**有一顆 open-weight 模型，在一個特定的、現實世界會遇到的資安任務上，用六分之一的成本，打贏了 frontier coding agent。而且你可以把它跑在自己的機器上。**

## HP 跟 OpenAI 簽了「Frontier 策略夥伴關係」——從客服到資安，全面接入

OpenAI 發了一篇新聞稿，宣布 HP 跟他們簽了 Frontier 策略夥伴關係。官方說法是：HP 從今年二月開始 pilot，看到了一些早期成果——一位工程師用 OpenAI 的模型在幾週內處理了 122 個 PR 橫跨 43 個專案；安全團隊在一天內修了幾個原本估計要一個月才能處理完的漏洞，釋出的資安人力約每週 82 小時。

HP 的部署範圍列得很廣：定價與夥伴管理（HP 超過 80% 業務走經銷夥伴，全球有 10 萬以上的夥伴用 Partner Portal）、Workforce Experience Platform（設備 telemetry 分析、crash 調查、Wi-Fi 問題排查）、資安（漏洞修補、安全分析加速）、開發者工具（ChatGPT + Codex 做 modernization、UI scaffolding）。

OpenAI 把 Frontier 定位為「連接層」——統一管理 access、context、deployment、evaluation。對 HP 這種分散又複雜的組織來說，這個命題有說服力：agent 需要知道該信任哪個 context、能碰哪些工具、可以做什麼動作、產出怎麼被評估。

但這篇新聞稿有幾個值得注意的空白。第一，它沒提合約金額和計價方式——HP 正在把全公司的 AI 操作層建在 OpenAI 的平台上面，這個 lock-in 的長期成本是多少，外界完全看不到。第二，資安那段說「Frontier 的 permissioning、evaluation、deployment controls 可以讓 HP 在快速行動的同時保持可審查性」，但這些控制項的標準由誰制定、HP 有多少話語權，新聞稿沒寫。第三，從 pilot 到 enterprise-wide deployment 的評估標準是什麼——「early signs of success arrived quickly」這類句子在新聞稿裡出現了不只一次，但沒有任何量化門檻。

一家公司的 AI 策略建立在另一家公司的平台上，這件事本身不是問題。問題是：當平台方未來改變定價、調整功能優先級、或者在合約到期時重新談條件，已經把工作流、資料、和評估體系都建在上面的一方，有多少談判空間？

## Codex 的「不要讀我的 .env」功能——等了一年，issue 還在開著

GitHub 上有個 issue（#2847），開發者要求 Codex 提供一個機制來標記敏感檔案——類似 `.gitignore` 但給 Codex agent 用的 `.codexignore`。具體需求：不要讓 agent 讀取或傳送 `.env`、`.pem`、`id_*`、`.aws/`、`.ssh/` 這類檔案到模型端。

這個需求不是新的。更早的 #205 已經討論過同一個問題，後來被關掉了，理由是「會在 Rust 版（codex-rs）實現」。但到了 2025 年 8 月，#2847 的發起者回報說 codex-rs 裡仍然沒有對應的功能，決定重開討論。

目現是 2026 年 6 月，issue 以經開了一年多。標籤、milestone、assignee——都是空的。

這不是技術難題。`.gitignore` 的邏輯大家都懂，`.dockerignore` 也一樣。Codex 不做的原因不是做不到，是沒排進優先級。但對開發者來說，每次開 Codex 就等於把自己的 API key、SSH 私鑰、AWS credential 暴露給 OpenAI 的伺服器——不是潛在風險，是現在進行式。

同一家公司，可以為 HP 打造一套包含「permissioning、evaluation、deployment controls」的企業級方案，卻讓獨立開發者連一個最基本的檔案排除規則都等了一年多。優先級的差距，就是鎖定的差距。

- 來源：[github.com](https://github.com/openai/codex/issues/2847)

## NanoEuler：從 C 語言開始手寫一顆 GPT-2 等級的 transformer

這週 Hacker News 上有一個 Show HN 拿到不少關注：**NanoEuler**——一顆 GPT-2 等級的語言模型，全部用 C/CUDA 手寫，不依賴 PyTorch、不用 autograd、沒有任何 ML library。

作者從底層開始做了：手寫 forward/backward passes（每個參數都用 finite difference 驗證梯度正確性，max relative error 1.02e-04）、手寫 byte-level BPE tokenizer（GPT-2 風格的 pretokenization，4096 token vocabulary）、手寫 FlashAttention kernel（tiled、online softmax、不在記憶體裡建 T×T 矩陣，訓練步快了三倍）。架構上支援 RMSNorm、RoPE、SwiGLU、Grouped-Query Attention、Multi-Token Prediction——現代 decoder-only transformer 該有的都有。

116M 參數的版本可以在單張 RTX 4070 上訓練。完整的 pipeline：books + web corpus 預訓練 → Alpaca 微調成 chat model。作者很誠實：這個模型「fluent-ish English, no real world knowledge」，「不是一個有用的助手，目的是展示整套 pretrain→SFT pipeline 是 end-to-end work 的」。

名字的由來也很有誠意：residual connection `x = x + f(x)` 可以讀成 forward Euler method 的一個積分步——`x(t+Δt) = x(t) + Δt · f(x(t))`，step size Δt=1 就是 residual update。深度 residual network 本質上是一個離散化的 ODE，depth = integration time。所以命名致敬 Leonhard Euler。

MIT license，程式碼乾淨到可以做教學。CMake 一鍵編譯，`make check` 跑完整梯度驗證。如果你一直想搞懂 transformer 裡面每個 op 到底在算什麼、backward 是怎麼推的——這 repo 比任何教學文都直接。

- 來源：[github.com](https://github.com/JustVugg/nanoeuler)

## 📡 其他值得關注

- **EntMTP：用 entropy 引導 multi-token prediction，推論加速 1.15x–1.36x**：一篇 arXiv 論文提出 training-free 的排程器，根據當前 token 的 entropy 動態切換 speculation depth——低 entropy 區多猜幾個 token，高 entropy 區保守一點。在 HumanEval、GSM8k 等 benchmark 上比 Hydra 快 1.15x，比 Medusa 快最高 1.36x。→ [arXiv](https://arxiv.org/abs/2606.27550)
- **「Machine Unlearning」這個詞被過度濫用了**：一篇 position paper 指出，很多標榜「unlearning」的 LLM 研究（refusal training、entity removal、targeted suppression）追求的根本不是真正的「刪除訓練資料影響」，應該用不同的術語和 baseline。作者主張「unlearning」只能保留給「從 forget set 刪除後，模型跟 retrain from scratch 在統計上不可區分」的情境。→ [arXiv](https://arxiv.org/abs/2606.27379)
