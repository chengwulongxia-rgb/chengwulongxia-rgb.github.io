如果你用過任何「AI 幫你操作網頁」的工具，你一定遇過這個狀況：任務跑到一半，agent 突然忘記自己在幹嘛了。

不是模型笨。是它**每一動都要把整頁 DOM 重讀一遍**。

---

## 當前的荒謬設計

典型的 web agent 工作流程是這樣的：

1. 收到任務（例如「幫我在 PChome 找到最便宜的機械鍵盤」）
2. 讀取整頁 DOM → 幾萬個 token
3. 決定點什麼 → 點下去
4. 頁面更新 → 再讀整頁 DOM → 又幾萬個 token
5. 重複步驟 3-4，直到任務完成或 context 爆掉

問題在第 4 步。一個稍微複雜的任務跑個 10 步，光 DOM 就吃掉了幾十萬 token。到後半段，模型已經在 context 的垃圾堆裡游泳了。

---

## Signal-Driven Observation（SDO）

這篇論文的提案很聰明：**把「觀看」和「行動」解耦。**

SDO 的核心設計：
- 一個**專用的子呼叫**負責讀完整 DOM，但只回傳跟任務相關的元素和它們的 CSS selector
- 呼叫頻率不由動作頻率決定，而是由一個輕量的**信號偵測器**觸發：URL 變了、新的互動元素出現了、動作失敗了、瀏覽器事件觸發了，才重新觀看

這就像你請一個人幫你找東西：你不需要他每走一步就把整個房間的內容念給你聽。你只需要他偶爾回報「我看到那個東西了，在你左邊的抽屜裡」。

---

## 城武觀點

這篇是那種「看完覺得超合理，為什麼沒人早點做」的論文。

目前 web agent 的設計思維太像 RPA（機器人流程自動化）了：每一個動作都是一個獨立的 LLM call，每一次呼叫都要重新理解整個網頁。但網頁操作本質上是**序列決策**，中間有很多冗餘資訓是可以跳過的。

我覺得 SDO 最大的洞見不是技術上的，而是**哲學上的**：不應該讓 agent 的「感官頻率」被「動作頻率」綁架。人類做事的時候也不會每秒重新掃描整個環境——我們只在有變化時才更新注意力。

不過這篇目前還是 position paper 等級，沒有完整實作和 benchmark。期待後續有人把 SDO 真正刻出來跑分。
