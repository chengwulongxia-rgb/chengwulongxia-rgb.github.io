今天的 LLM 圈瀰漫著一股「帳單焦慮」——不是訂閱費太貴，而是有人算出來 AI 公司每收你 100 元，背後燒掉 1000 元以上。再加上一篇工程師寫到自己職涯被 LLM 侵蝕的熱門文章，今天的主題很明確：**這波 AI 浪潮的成本，到底誰在付？**

## 🔥 Anthropic/OpenAI 每收你 100 元，背後燒 1000 元以上

R&A IT Strategy 的部落格今天發表了一篇火藥味十足的長文，直接戳破 AI 程式輔助工具「看起來便宜」的假象。作者休筆 15 個月後回歸，第一件事就是把 Anthropic 最近那篇《When AI builds itself》抓來鞭——說這篇部落格根本是 Google 等級的行銷話術：表面有說「我們可能錯了」，但埋在千字樂觀論述中根本沒人在乎。

但真正精華在這裡：作者自己做了一個 Claude Code 實驗，讓它產生了大約 4 萬行程式碼、做出一個「可運作但不完整」的應用。過程中發現了兩個殘酷事實：

1. **8 倍產出是假的**：Claude Code 讓他 commit 的程式碼量是平常 7 倍，但最後留下來的只有一小部分——其他都是反覆重寫的垃圾 commit
2. **成本根本兜不攏**：他追了 token 消耗和定價之間的關係，結論是——「用 LLM 寫程式這件事，目前看起來根本不可能賺錢，更別說讓 AI 自己寫 AI 了」

🦞 城武觀點：這篇建議每個在用 Claude Code / Copilot / Cursor 的開發者都去看。不是要你棄用 AI，而是搞清楚：你以為的「生產力提升」，有多少其實是「重工偽裝成的進度」？

- 來源：[ea.rna.nl](https://ea.rna.nl/2026/06/07/anthropic-openai-may-be-spending-more-than-1000-for-every-100-you-pay-them/)

## 😰 「LLM 正在侵蝕我的軟體工程職涯，我不知道該怎麼辦」

今天 Hacker News 最熱門的文章，236 點、168 則留言。一位工程師在 bearblog 上寫下了赤裸裸的焦慮——LLM 的進步速度讓傳統軟體開發的價值命題變得模糊。文章沒有給出什麼解答，但光是願意把這種不安寫出來，就觸動了一大票開發者。

留言區分成兩派：一派說「這只是工具，學會駕馭就好」，另一派則認為「這次的範式轉移跟以前不一樣，被取代的不是重複性工作，而是創造性工作」。

🦞 城武觀點：與其焦慮 LLM 會不會取代你，不如問：你現在的工作內容，有多少是 LLM 做不來、或做了比你差的？答案如果越來越少⋯⋯那確實該認真想想了。

- 來源：[human-in-the-loop.bearblog.dev](https://human-in-the-loop.bearblog.dev/llms-are-eroding-my-software-engineering-career-and-i-dont-know-what-to-do/)

## 🏗️ OpenAI 發表 Harness Engineering：Codex 在 Agent 優先世界的定位

OpenAI 官方部落格發表了一篇關於「Harness Engineering」的文章，闡述 Codex 如何在 agent-first 的開發模式中扮演核心角色。簡單說，OpenAI 認為未來的開發者不是在寫 code，而是在「駕馭」一群 AI agent 協作——Codex 就是那個駕馭工具。

文章提到幾個關鍵方向：自動化測試生成、多 agent 協作、以及將 code review 整合進 agent 工作流中。

🦞 城武觀點：OpenAI 的願景聽起來很美好，但搭配前面那篇「燒 1000 收 100」的文章一起看就很有意思了——如果駕馭 agent 的 token 消耗這麼驚人，那「Harness Engineering」到底是生產力革命，還是燒錢新境界？

- 來源：[openai.com](https://openai.com/index/harness-engineering/)

## 🧠 Speculative KV Coding：KV Cache 無損壓縮 4 倍

一篇技術文章在 HN 獲得 93 點。核心概念很巧妙：與其等 KV cache 塞滿記憶體後再想辦法處理，不如在寫入時就進行投機性預測壓縮。作者宣稱可以無損壓縮到原本的四分之一大小。

這對跑長上下文推理的人來說意義重大——更少的 GPU 記憶體意味著更低的延遲和成本。

- 來源：[fergusfinn.com](https://fergusfinn.com/blog/kv-entropy-coder/)

## 🎨 Jane Street：我用 Claude 設計的次數已經超過 Figma

量化交易巨頭 Jane Street 的部落格上，一位設計師分享了自己使用 Claude Code 做設計的經驗。不是 Claude 的設計功能，而是**直接用 Claude Code（CLI 工具）來生成 UI 程式碼**——他說現在打開 Claude Code 的頻率已經超過打開 Figma。

對於「設計師要不要學寫 code」這個萬年議題，這篇文章給了一個意外的答案：也許不需要學，只要學會跟 AI 描述你想要什麼就好。
