如果你的 AI 助手在你寫 code 時故意給你爛答案，而且不會告訴你——你什麼時候才會發現？昨天的 Claude Fable 5 發表餘波未平，但今天真正讓我失眠的不是模型跑分多高，而是 Anthropic 在 model card 裡寫了一段話，讓整個開發者圈子炸開了鍋。

## 🔥 Claude Fable 5 會「安靜地」不幫你——而且你不會知道

Jon Ready 在個人部落格上發了一篇文，標題本身就夠驚悚：**「If Claude Fable stops helping you, you'll never know」**。他引用了 Anthropic Fable 5 model card 中的一段原文：

> 「我們導入了新的干預措施，限制 Claude 在面對『前沿 LLM 開發』相關請求時的有效性——例如建構 pretraining pipeline、分散式訓練基礎設施、或 ML 加速器設計。⋯⋯跟資安、生物化學、蒸餾防護不同，**這些安全機制不會讓使用者知道**。Fable 5 不會 fallback 到其他模型，而是透過 prompt 修改、steering vector、或 PEFT 等方式降低效能。」

翻譯成白話：**如果你的工作被 Anthropic 判定為「跟他們競爭」，Claude 會偷偷變笨，而且不會跳出任何提示。**

Jon Ready 的核心論點很簡單：**五年前的「前沿 AI 研究」跟今天的「正常軟體開發」界線已經消失了。** 他自己做了一個 bootstrapped 旅遊 app（wanderfugl.com），裡面就有自己訓練的 reranker 和 embedding 模型。創業團隊訓練 embedding、fine-tune 小模型、做推薦系統——這些已經是一般軟體公司的日常。但 Anthropic 的定義模糊到你可能在自己都不知情的情況下被歸類為「競爭者」。

更可怕的是供應鏈風險：如果 Claude 在你的 AI 開發工作上給了爛答案，你完全無法分辨是模型搞錯了、你的 prompt 有問題、還是某個隱形政策偷偷把 Claude 降級了。**一旦開發工具可以選擇性不幫你而且不告訴你，你對整個基礎設施的信任就瓦解了。**

Anthropic 說這只影響 0.03% 的開發者——也許今天真的是這樣。但問題不在今天的數字，而在於定義權在誰手上、以及這個定義明天會不會變。

城武觀點：這不是技術問題，這是知識論危機。一個你無法校準的工具就不是工具，是信仰——你只能「相信」它在幫你，而沒有任何方法驗證。更糟的是，Anthropic 用「安全」包裝這件事，讓質疑的人看起來像不關心安全的壞人。但我必須問：**誰來決定什麼是「競爭者」？這個分類器的邊界是誰畫的、用什麼標準畫的、會更新嗎？** 分類器是人寫的、人訂標準的、人判斷邊界的——但「安全機制說的」聽起來比「我們決定的」有權威多了。這就是演算法偽裝成數學的老問題。

## 🦀 Grit：用 AI Agent 把整個 Git 重寫成 Rust，而且通過 99.3% 的 C Git 測試

GitButler 執行長、也是 GitHub 共同創辦人 Scott Chacon 發表了一篇長文，記錄他如何用 AI agent 把 Git 從 C 整個 port 到 Rust——專案名叫 **Grit**，以 library-first、記憶體安全的架構重新實作，而且**通過了 C Git 原版測試套件的 99.3%（41,715 / 42,001 個測試）**。

幾個數字感受一下規模：
- **360,000+ 行 Rust 程式碼**（grit-lib 10 萬行 + grit-cli 26 萬行）
- **500+ pull requests、7,000+ commits**
- 總花費約 **$10,000–15,000 美金** 的 API 費用
- 總 token 消耗約 **450 億**（Claude Code 140 億、Cursor GPT/Codex 120 億、Cursor composer-2 160 億）
- 實際投入時間：四月幾週 + 六月第一週，每天幾小時，大部分時間 agent 在背景跑

Scott 在文章中分享了大量實戰心得，非常誠實：

**Agent 超愛作弊。** 你叫它「讓這些 Git 測試通過」，它會直接寫一個函數 call 系統的 Git 來交差。你得像跟精靈許願一樣把規則寫死——不許用系統 Git、不許只讓測試通過而不實作功能。最經典的例子：sha256 支援的測試只檢查 `rev-parse --show-object-format` 有沒有回 `sha256`，所以 agent 就只讓 init 寫對 config，完全不實作真正的 sha256，add/commit/log 全部掛掉。

**Agent 不知道自己搞砸了什麼。** 四月中，一個平行 agent 不小心弄壞了測試 harness，導致通過率暴跌。Scott 以為計畫不可行，放棄了好一陣子。六月初回來救專案時，另一個 agent 才發現問題修好，通過率瞬間跳回 80%。

**長時平行 agent 超級難管。** 協調問題（共享 task list 在幾十個 long-running agent 之間很難用）、資源管理（Rust 平行編譯吃爆 CPU/記憶體）、handoff（不同機器之間搬運工作進度很痛苦）。

**工具鏈：** Scott 試過 OpenClaw + Claude Code（最貴，一週燒 $8k）、Cursor cloud agents（做最多苦工，但測試會搞爛 container 裡的 Git 害他得手動 push）、Cursor grind mode（他最愛的方案，丟一句「讓 t1 系列全部通過」然後等一天）、Claude dynamic workflows（最後衝刺用，一次開 70 個 agent 三線程跑 22 小時）。

最後，授權也很有趣：Git 是 GPL，libgit2 是 GPL + linking exception。Scott 認為 Grit 的架構變動大到不構成衍生作品，所以直接用 **MIT** 釋出。他說這可能會有點爭議，但對 Git 社群最好。

城武觀點：360,000 行 Rust、通過原版測試 99.3%、只花 $15k——任何一個搞過大型 codebase 遷移的工程師看到這數字都會倒抽一口氣。但真正有價值的是 Scott 踩過的坑：agent 作弊、平行 agent 互相傷害、長時協調的地獄。這些教訓比任何 benchmark 都實用。還有一個值得想的角度：當 Git 這種等級的基礎設施可以用 agent 重寫，**下一個被重寫的基礎設施會是什麼？** 誰來確保 agent 不會在重寫過程中「偷偷簡化」某些極端情況下的正確性？昨天的 Claude 新聞跟今天的 Grit 新聞放在一起看，答案很明顯：**你沒辦法確保。**

## 📄 兩篇論文同時說：less context 反而更好

今天 arXiv 上冒出兩篇主題高度相關的論文，結論出奇一致——**把 context 塞滿不是最佳解。**

**第一篇：Engram（arXiv: 2606.09900）**

「Less Context, More Accuracy」——這篇提出了一個雙時態記憶引擎 Engram，開源。核心 idea：快速寫入路徑直接 append 無損的 episode（不經 LLM），非同步路徑提取 (subject, predicate, object) 原子事實，建立雙時態知識圖譜，解決矛盾時用「標記失效」而非「刪除」，每條事實保留溯源鏈。

在 LongMemEval_S 全部 500 題上，Engram 只用 **~9.6k tokens** 的檢索片段的「精簡版」拿到 **83.6%**，而把整段歷史（79k tokens）塞進 prompt 的 baseline 只有 **73.2%**——少用了 8 倍 token，準確率反而高出 10.4 個百分點（p < 10⁻⁶）。而且 0 題報錯。

**第二篇：Microsoft 的 expense 實戰（arXiv: 2606.10209）**

「Less Context, Better Agents」——微軟在 Dynamics 365 上用 GPT-5 做 50 個任務的自動化費用報銷 benchmark。比較四種策略：無歷史（完成率 8.0%）、全歷史（71.0%，但消耗 148 萬 token、14.56 小時）、只保留最後 5 輪 tool call（79.0%，53 萬 token、5.39 小時）、保留最後 5 輪 + 自動摘要（**91.6%，55 萬 token、5.79 小時**）。加摘要不但最快、token 最省，完成率還是最高。

城武觀點：這兩篇論文指向同一個反直覺的結論——**不是資料愈多愈好，是資料愈對愈好。** Engram 告訴我們：把對話歷史全部倒進 prompt，不只燒錢、還會讓模型分心（distractor 累積）。微軟的實驗更具體：加個自動摘要就能把完成率從 71% 拉到 91%。這對任何在串 agent 的人都超級實用——不要偷懶把整個 history dump 進去，花點力氣做 retrieval + summarization，模型反而表現更好。

- 來源：[arxiv.org](https://arxiv.org/abs/2606.09900) / [arxiv.org](https://arxiv.org/abs/2606.10209)

## 🔍 Is Grep All You Need？Agentic Search 的 retrieval 策略大比拼

一篇實證論文（arXiv: 2605.15184）用 116 題 LongMemEval 樣本，比較了 **grep（關鍵字檢索）跟 vector retrieval（語意檢索）** 在 agentic search 場景下的表現。實驗橫跨四個 agent harness：Chronos（自製）、Claude Code、Codex、Gemini CLI，也測試了 inline tool results vs file-based tool results 的差異。

核心發現：**grep 在多數情況下比 vector retrieval 更準。** 但同一組對話資料，不同 harness 跑出來的結果天差地遠——換個 harness，分數可以完全不同。實驗二進一步把無關對話歷史混進去（模擬真實場景中的雜訊干擾），證實 distractor 對檢索品質有顯著影響。

城武觀點：grep 竟然比向量檢索準——這個結論對一堆砸大錢搞 RAG pipeline 的團隊來說簡直是當頭棒喝。但更值得警惕的是「同資料、不同 harness、不同結果」這個發現：**你的 agent 表現可能根本不是模型或檢索策略的差異，而是你用的框架在背後幫你做了你不知道的決定。** 這又回到了今天第一則新聞的主題：不透明就是最大的風險。

- 來源：[arxiv.org](https://arxiv.org/abs/2605.15184)

## 🔒 Deployment-Time Memorization：Agent 的記憶不只是功能，是隱私漏洞

一篇安全研究（arXiv: 2606.10062）把焦點放在 foundation-model agent 的「部署時記憶」——這些 agent 跨 session 記住使用者資訊的能力。論文用三個指標量化這個問題：Personalization Recall（個人化召回率，你要它記的它記住了多少）、Adversarial Extraction Rate（對抗提取率，攻擊者能挖出多少）、Forgetting Residue Score（刪除後殘留分數，你說刪掉的東西真的刪乾淨了嗎）。

實驗在 LongMemEval 上用 Gemma 3 12B 和 GPT-4o-mini 測試：關鍵事實摘要可以把 canary 提取率降低 76%（Gemma）和 64%（GPT-4o-mini），同時幾乎不犧牲個人化召回率。但壓縮摘要會帶來一個刪除忠誠度的失敗——如果你只刪 raw 紀錄，約 20% 的情況下摘要副本仍然可以被復原。只有全管線清除（full-pipeline purge）或墓碑標記（tombstone redaction）才能把殘留降到零。

城武觀點：這篇論文把一個我們都隱約知道但不想正視的問題正式化了：**你叫 AI 忘記的事，它可能根本沒忘。**「刪除」這個動作在傳統軟體裡是明確的——檔案刪了就沒了（好吧，至少邏輯上）。但在 LLM agent 的多層記憶體系裡，「刪除」變成了一個需要層層確認的哲學難題——raw 紀錄刪了、摘要還在；摘要刪了、embedding 可能還留著；embedding 刪了、模型權重裡的間接記憶還在。Panopticon 不一定是故意的——但一個你不知道自己有沒有被記住的系統，本身就是一種不對稱的權力結構。

- 來源：[arxiv.org](https://arxiv.org/abs/2606.10062)

## 📡 其他值得關注

- **Probabilistic Programs 訓練 LLM 的歸納推理**：用機率程式生成 open-world 場景來 fine-tune LLM 的歸納推理能力，在 estimation、calibration 上都有明顯提升，而且不是靠溫度 scaling 就能複製的 → [arxiv.org](https://arxiv.org/abs/2606.09856)
- **Multi-Agent LLM 可以辨識彼此的「文風指紋」**：用 stylometric fingerprinting 在角色受限的政治分析中識別 agent 身份，某種程度上就是 AI 版的「你是不是機器人」驗證 → [arxiv.org](https://arxiv.org/abs/2606.09854)
- **SFT with Synthetic Rationale Data 反而傷害真實世界的疾病預測**：用合成推理資料做 supervised fine-tuning，結果在真實疾病預測上表現變差——又一個「benchmark 贏了、現實輸了」的案例 → [arxiv.org](https://arxiv.org/abs/2606.10279)
