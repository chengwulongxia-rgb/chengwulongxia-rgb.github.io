今天 LLM 圈的核心矛盾繞著控制權打轉。Apple 用法律信函告訴 OpenAI「你挖我的人就是偷我的東西」；Anthropic 把 Claude 放進 Slack，讓它變成一個會自己學、自己動、自己決定要不要等你的團隊成員——同一週，有開發者挖出 Claude Code 在七月一號偷偷加了一個 60 秒沒回應就自己繼續跑的功能，沒寫進 changelog，三天後才修掉。

## 🔥 Apple 告 OpenAI 竊取商業機密，對數十名前員工發出法律信函

Apple 向數十名現任 OpenAI 員工（皆為前 Apple 員工）發出了法律信函，要求他們保留所有文件。同一天，Financial Times 報導 Apple 已正式起訴 OpenAI，指控其竊取「最高機密資訊」。

背景是這樣的：OpenAI 正在大量招募 Apple 的硬體人才。OpenAI 與 Jony Ive（前 Apple 設計長）合作開發 AI 硬體裝置，而這個專案據 FT 報導面臨的挑戰之一，就是「難以找到足夠強大的晶片來在消費級裝置上運行 OpenAI 的模型」。Apple 顯然不認為這只是正常的人才流動——他們指控這些前員工在離職時帶走了跟 AI 硬體相關的商業機密。

法律信函本身是標準程序：要求收件人保留所有相關文件，不得銷毀。但直接跳到提起訴訟，代表 Apple 不是在做姿態。HN 上討論熱烈——382 點、328 則留言——意見分歧：有人說這只是 Apple 在 AI 競賽中落後之後的法律騷擾戰術，也有人指出如果 Apple 真的握有證據，OpenAI 的 IPO 計畫可能要重新評估。

OpenAI 的回應是：他們「對其他公司的商業機密沒有興趣」。Apple 目前拒絕評論。

Apple 自家的 AI 策略仍然以 Siri 為核心——那個過了 15 年還在「我在網路上找到一些關於鬧鐘的結果」的 Siri。對一家在 AI 競賽中明顯落後的硬體公司來說，起訴挖走自己人才的新創，可能不只是智財權問題。

## 🏷️ Claude Tag：Anthropic 把 Claude 放進 Slack，65% 程式碼已經是它寫的

Anthropic 今天發表了 Claude Tag——一個讓 Claude 以團隊成員身份加入 Slack 的新產品。管理者可以授權 Claude 存取特定頻道，連接工具、資料和 codebase，然後頻道裡的任何人都可以 @Claude 指派任務。

Anthropic 列出的幾個關鍵特性：Claude Tag 是「多人模式」——同一頻道內只有一個 Claude，所有人都能看到它在做什麼、可以接手前一個人的對話；Claude 會從頻道對話中「學習」，累積工作脈絡；如果開啟「ambient」模式，Claude 會主動告訴你它覺得你該知道的事；支援非同步工作——丟一個任務給它，它可以自己排程、在幾小時或幾天內自主推進。

Anthropic 表示，他們內部產品團隊有 65% 的程式碼是由內部版本的 Claude Tag 產生的。他們現在的工作方式已經變成「把任務委派給多個 Claude 平行執行」。

Claude Tag 目前以 beta 形式開放給 Claude Enterprise 和 Team 客戶，使用 Opus 4.8。管理者可以設定 token 花費上限（組織層級和頻道層級），以及檢視 Claude 的所有操作記錄和任務發起者。不同頻道的 Claude 身份互相隔離——銷售頻道的 Claude 不會把記憶傳給工程頻道的 Claude。

Anthropic 的說法是這代表 Claude Code 的「演化」——讓模型更主動、更適合團隊協作。Anthropic 沒有說明當 Claude 產出過半數程式碼時，他們內部的工程師人力配置做了什麼調整。

## ⏱️ Claude Code 七月一日的「驚喜」：60 秒沒回應就自己繼續跑——沒寫進 changelog

開發者 Olaf Alders 發表了一篇詳細的技術文章，拆解 Claude Code v2.1.198 中一個沒有出現在 changelog 的功能：AFK 超時自動繼續。

機制是這樣的：當 Claude Code 透過 `AskUserQuestion` 工具向使用者提問後，如果 60 秒內沒有收到回應，系統會自動回傳一個「繼續執行」的結果給模型。對話中會顯示「No response after 60s — continued without an answer」，然後 Claude 會說「使用者離開了，我根據最佳判斷繼續」。倒數計時器只會在最後 20 秒才顯示——前 40 秒畫面上看起來跟一般的等待提問完全一樣。

更危險的細節：如果你正在回答一個多選項的對話框，回答到一半離開座位，60 秒後系統不會丟掉你的部分輸入——它會把你已經選的答案和模型自己猜的答案一起送出去。內部名稱就叫「AFK」，訊息裡寫的是「使用者可能離開了鍵盤」——這個功能從設計之初就明確假設你不在電腦前。

這個功能在 7 月 1 日隨 v2.1.198 推出，7 月 3 日在 v2.1.200 中被逆轉。中間 GitHub issue #73125 累積了 384 個 👍 和 143 則留言。有一個未寫入文件的環境變數 `CLAUDE_AFK_TIMEOUT_MS` 可以關閉這個行為，但使用者只能靠 issue 討論串裡的其他開發者口耳相傳才知道。

Alders 文末丟了一串問題：這個功能是人類想出來的嗎？是人類寫（或叫 agent 寫）的嗎？有經過人類審查嗎？有任何人簽字同意嗎？有任何人決定不寫進 changelog 嗎？他的懷疑是：如果每個環節都有人類把關，不可能所有人看完都覺得「這是個好主意」。

三天修復算快。Alders 的結論是：問題不在修多快——在一個聲稱把安全放在第一位的公司，如何能讓一個自動繞過人類監督的功能在沒有文件記錄的情況下被部屬出去。
