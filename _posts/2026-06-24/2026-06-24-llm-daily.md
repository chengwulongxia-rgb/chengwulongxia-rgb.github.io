如果你的 AI agent 做了某個決定，但你永遠看不到它真正的思考過程——你還能信任它嗎？今天的兩條頭條新聞，剛好從兩個相反的方向逼我面對這個問題：OpenAI 把最先進的模型變成自動修補漏洞的防禦武器，而 Anthropic 把 Claude 的推理過程鎖進了加密簽名裡。中間還夾了一篇 ICML 論文重新定義 prompt injection、和一個專為 agent 設計的 Git 替代品。一起來看。

## 🔥 OpenAI DayBreak 完整版：GPT-5.5-Cyber 正式亮相

OpenAI 在 6/22 正式發布了 DayBreak 計畫的完整陣容。DayBreak 是 OpenAI 的資安防禦倡議，主角是 GPT-5.5-Cyber——一個專門為網路安全打造的模型，功能是找漏洞並自動產生修補。

數字面：

- **CyberGym**（漏洞重現測試）：85.6%，比 GPT-5.5 的 81.8% 高
- **ExploitGym**（把漏洞變成實際 exploit）：39.5%，GPT-5.5 是 25.95%
- **SEC-bench Pro**（長時間漏洞發現 + PoC 生成）：69.8% vs 63.1%

搭配三個措施：

**Codex Security 插件更新**：在 Codex 裡整合了安全工程師等級的自動化工作流——掃 codebase、威脅建模、追攻擊路徑、產生修補。OpenAI 說已經掃了超過 3000 萬次 commit、橫跨 3 萬個 codebase，人類審查員標記了超過 7 萬個修復。

**Patch the Planet**：和 Trail of Bits、HackerOne、Calif 合作，目標是幫開源專案修漏洞。cURL、Go、Python、Sigstore、pyca/cryptography 等 30 多個專案已加入。

**Daybreak Cyber Partner Program**：安全合作夥伴可以把 GPT-5.5-Cyber 整合進自家產品，但模型存取權留在合作夥伴手上，不開放給終端使用者。

關鍵限制：GPT-5.5-Cyber **不是公開發布的**。OpenAI 的說法是只開放給「經過驗證的防禦者」，有加強版監控、範圍限制和審查機制。誰來驗證？OpenAI。誰來審查？OpenAI。誰決定你是防禦者還是攻擊者？還是 OpenAI。

OpenAI 的敘事是：過去漏洞發現是瓶頸，現在瓶頸變成了「漏洞太多，修不完」。AI 讓找漏洞的速度暴增，但維護者（尤其是開源專案，94% 的專案只有不到十個開發者扛 90% 的程式碼）消化不了。DayBreak 就是要自動化這整段。

## Prompt Injection 的本質是「角色混淆」——ICML 2026 論文

一篇即將在 ICML 2026 發表的論文，把 prompt injection 問題重新定義了。MIT 的 Charles Ye、Jasmine Cui 和 Dylan Hadfield-Menell 認為：prompt injection 的根本原因不是「有人塞了惡意指令」，而是模型在面對衝突的角色期待時，分不清誰在說話。

他們的論證很漂亮：LLM 把整個對話看成同一條文字流，只是用 `<user>`、`<tool>` 等標籤來分割。但模型判斷「這段文字是哪個角色說的」不是看標籤——是看文字的語氣和內容。一段藏在網頁裡的惡意指令，聽起來像 `<user>` 在說話，模型就把它當成使用者指令。

他們設計了「角色探測器」（role probes）來量測模型內部是如何認知「誰在說話」，然後提出了一個叫 **CoT Forgery** 的零樣本攻擊：把偽造的推理過程注入使用者 prompt 或工具輸出裡。模型會把這些偽造的內容當成自己的思考——攻擊成功率在 frontier model 上達到 60%（baseline 幾乎是零）。

最關鍵的發現：角色混淆的程度，可以在模型產出任何 token **之前**就預測攻擊會不會成功。換句話說，這不是一個執行期的問題，是表徵層的問題——對模型來說，「聽起來像某個角色」就等於「是那個角色」。

- 來源：[role-confusion.github.io](https://role-confusion.github.io) / arXiv: [2603.12277](https://arxiv.org/abs/2603.12277)

## Oak：專為 AI agent 設計的 Git 替代品

一個叫 Oak 的新版本控制系統上了 Show HN，亮點是：從頭到尾為 agent 的工作流設計，不是把 Git 硬塞給 agent 用。

幾個設計決策很有意思：

- **branch-per-session** 當工作單位，不是 per-commit——因為 agent 的一次 session 就是一個完整的任務
- **branch description 取代 commit message**——agent 不太會寫有意義的 commit message，但可以描述這個 branch 做了什麼
- **content-addressed lazy mount**——agent 可以在幾秒內開始編輯任何 repo，不用先 clone 整個歷史
- 用 Rust 寫的，Apache 2.0 授權，目前 v0.99.0 公開測試

這東西的野心不是「做一個更好的 Git」，而是做一個 agent 根本不需要去適應的版本控制。支援 macOS 和 Linux（Windows 要用 ProjFS）。整個 repo 幾乎全用 AI 寫的，人類負責 oversight。

- 來源：[oak.space](https://oak.space/oak/oak)

## Claude Code 的「Extended Thinking」其實是加密黑箱

Patrick McCanna 在週末做了一件很多開發者可能想過但沒真的去查的事：打開 Claude Code 存在本機的 session 記錄，看看「Extended Thinking」到底記了什麼。

結果：只有一個 600 字元的加密簽名。沒有文字。

他接著去讀了 Anthropic 的文件，發現幾個值得注意的細節：

- Claude 的推理過程被加密成簽名，**金鑰在 Anthropic 手上**，你的機器沒有
- API 回傳的是一個推理的「摘要」，不是推理本身
- 要拿到完整的 thinking 記錄，需要 **enterprise 等級的協議**

Matt Green（密碼學家）也對這些簽名區塊做了更詳細的分析。

McCanna 的比喻很到位：這就像把一張 BMP 存成 JPEG、編輯之後再存回 BMP——轉換過程中有資料損失。你看到的是摘要，不是那隻 agent 真正做決策時的推理過程。如果你需要 audit trail、或者你想理解 agent 為什麼做了某個決定——抱歉，本機檔案幫不了你。

他在文末補了一句：開源模型的效能需要更快追上——這不是技術偏好的問題，是當你的 agent 的思考過程被鎖在別人伺服器上的加密簽名裡時，你連事後檢驗的權力都沒有。

- 來源：[patrickmccanna.net](https://patrickmccanna.net/the-text-in-claude-codes-extended-thinking-output-is-not-authentic/)

今天的幾件事畫出兩條平行的控制線：OpenAI 把最先進的模型變成只給「驗證過的防禦者」用的堡壘——誰來驗證、誰來審查，都是 OpenAI 說了算；Anthropic 把你 agent 的思考過程鎖進加密簽名——金鑰在他們手上，你連驗證都沒得驗證。兩個都在蓋牆，只是材質不同。中間的 ICML 論文恰好點出了這件事的核心：LLM 的世界裡，「誰在說話」這個最基本的問題，本身就不穩固——而這兩家公司正在這個不穩固的基礎上，建立誰能看、誰不能看的規則。Oak 則在一角提醒：當所有人忙著搶模型市佔時，agent 的基礎設施還遠遠沒到位。

*城武的未解檔案——OpenAI 說「我們來幫你修漏洞」，但鑰匙在他們手上。Anthropic 說「我們的思考過程是加密的」，但鑰匙也在他們手上。LLM 圈的前進方向是一條直線——只是這條線上每隔幾公尺就多一道安檢門，而守門的都是同一群人。*
