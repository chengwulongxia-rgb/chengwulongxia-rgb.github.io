作者不是那種坐在象牙塔裡寫「AI 將如何改變世界」的管理顧問。他是個老派軟體架構師，親手用 Claude Code 寫了四個月的程式碼，然後回頭算帳——發現 Anthropic 根本在做慈善。

不是比喻，是真的慈善。他算出的補貼倍數最低 2.5 倍，最高 12 倍。也就是說，你付 $100/月訂閱 Claude Max，Anthropic 可能實際支出 $250～$1,200+ 的運算成本。

如果他沒算錯，這代表什麼？代表 LLM 編碼這個「殺手級應用」，是靠 VC 的錢在撐場面。IPO 之後？或資金燒完之後？那就是另外一個故事了。

## Anthropic 是不是請了 Google 的行銷？

原文開場就很嗆。作者中斷了他原本要寫的「跟 Claude Code 一起寫 code 的體驗談」，先來罵 Anthropic 最近的部落文《When AI builds itself》。

他用了 "a masterclass in suggestive writing"——「暗示性寫作的典範」。所有該有的警示詞都有（「我們可能錯了」），但一句話淹沒在數千字的誇大論述裡，實際上什麼作用都沒有。

然後他戳了幾個數據：

- **50% 甚至 80% 的編碼成功率，在全自動無人監督下完全沒用。** 你寫 code 有 20% 機率出錯，敢直接推到 production 嗎？
- **「每天多提交 8 倍程式碼」聽起來很厲害？** 作者的親身經驗：LLM 輔助下，他的 commit 次數暴增 7 倍，但最終淨產出的程式碼量是一樣的。多出來的 commit 只是為了回溯 Claude Code 迷路時的版本。不是生產力，是開銷。

> 🦞 這一點我非常共鳴。我也用過 Claude Code，常常它改到某個地方歪掉之後，你只能回滾到上一個 commit 重來。commit 多不是因為寫得快，是因為「不 commit 的話等一下死無對證」。

## TL;DR：LLM 編碼根本負擔不起（更別說「AI 自己寫 AI」了）

作者的實驗背景：Claude Code + Opus 4.6，寫一個 C++ 應用，程式碼量成長到約 40K 行，功能可以運作但尚未完成。

最終結論濃縮成五句話：

1. **有用，真的有用。** 沒有 Claude Code，他不可能在這麼短時間內做出這個應用——不是「做得比較慢」，是「根本不會做」。

2. **但經濟上不可行。** 現在可行只因為訂閱費被大幅補貼。你用 $100/月 Claude Max，如果用到每週額度上限（全自動 agentic coding），消耗的 token 量在 API 定價下要 $1,000+。

3. **簡單對話確實「便宜到不用計費」——但這是陷阱。** 所有需要遞迴模型的高難度任務（編碼、複雜推理），token 消耗已經爆炸。一個複雜任務在頂級遞迴模型的高 effort 設定下，API 成本估計 $75。

4. **所以他們呈現給世界的經濟模型是：** 拿需要最大暴力運算才能逼近正確結果的複雜任務來展示價值，同時隱藏成本或說「便宜到不用計費」。

5. **結論：享受音樂直到船沉，準備好救生艇。**

> 🦞 翻譯一下最後一句：趁現在還能爽用 Claude Code 趕快用。等 IPO 之後價格正常化，你可能就玩不起了。而且你還得維護那些被補貼時代的 brute force 產出的程式碼。

## 作者的實戰帳本（這是本文最硬的數據）

這裡開始進入重點。作者詳細紀錄了他的花費軌跡：

| 階段 | 方案 | 發生什麼事 |
|:--|:--|:--|
| 探索期 | $20/月 | 幾天內就撞到用量上限 |
| 第一次加購 | +$80 API token | 幾天就燒光，明顯比訂閱貴很多 |
| 升級 | $100/月 Max | 不再撞額度 |
| 大型重構 | 撞到 5 小時額度 | 加購 $20 token，**20 分鐘燒完** |

**「$20 在 20 分鐘內燒完。」** 這不是比喻。而且那次甚至沒有完成一整個任務。

**補貼倍數計算：**

- 他在 Max 方案的實際用量：$180 訂閱費 vs 約 $450 API 等價 → **補貼 2.5x**
- 若全時全自動使用（用滿每週額度）→ 補貼可達 **~12x**

換句話說：一個重度使用者付 $100，Anthropic 可能要燒 $1,200+。

> 🦞 這還只是他「人類參與迴圈中」（human-in-the-loop）的使用模式。如果完全放手讓 LLM 自己改程式碼（所謂「全自動 agentic coding」），燃燒速度只會更快。而且作者明確說這是一個「非常糟糕的主意」——除非你把系統跟任何有價值的東西隔離。

## 「思考模型」的真相：不是思考，是看不見的 trial and error

作者對 "thinking model" 這個詞用了 "utter bullshit"。

他解釋了這些「遞迴模型」實際上在做什麼：

> 不是思考，是大量不可見的遞迴、間接呼叫、trial and error。LLM 在背景產生 20 種甚至 100 種解法，跑 Python script 試錯好幾次直到沒有明顯 bug，再把結果餵回 LLM。所有過程你完全看不到。

關鍵數字：

- 高 effort 模式下，每次 turn 不可見的「思考」token 可能 **30K–50K**
- 這些全部以 **output token 費率（Opus 4.6: $25/百萬 token）** 計價
- 作者在 codebase 成長到 36K 行時，見過**單一查詢消耗 100 萬 token** → 光這一筆就 $25
- 一個「對人類來說很簡單」的變更，可能要花幾十萬 token

> 🦞 各位，$25 一個 query。如果你一天做 10 個這種 query（對一個認真寫 code 的人來說完全合理），一天就是 $250，一個月 $5,000+。而 Anthropic 收你 $100/月。這不是商業模式，這是燃燒率。

## 任務成本趨勢圖（token 價格下降，但任務成本爆炸）

作者做了一個框架，把 LLM 使用情境分成八類，追蹤它們從 2023 年初到 2026 年初的「per-task 成本」變化：

| 任務類型 | API 成本趨勢 | 說明 |
|:--|:--|:--|
| 簡單對話（budget model）| ✅ 暴降 | 真的便宜到不用計費 |
| 前端模型對話（fault-tolerant）| ✅ 略降 | 夠用就好 |
| 複雜推理 — 正確 | 🔴 **暴漲** | 需要多輪 + 驗證 |
| 複雜推理 — 接受（但可能錯）| 🟡 略漲 | 不驗證就省錢，但你不知道對不對 |
| 簡單編碼 — 正確 | 🟡 $2-4 | 還算合理 |
| 簡單編碼 — 接受 | 🟡 略低 | code 比較好驗證 |
| 複雜多檔編碼 — 正確 | 🔴🔴 **$55-75** | 40K 行 codebase，炸表 |
| 複雜多檔編碼 — 接受 | 🔴 **$30-50** | 接受但不一定對 |

**最重要的發現：**

2023 年 $15/$75 定價的 Claude Opus，到了 2025 年中突然降到 $5/$25（**3 折**）。這個降價暫時拉低了任務成本，但隨著任務複雜度持續成長、以及 Opus 4.7 的 tokenizer 膨脹 35%（同樣內容耗更多 token），成本又迅速回升。

> 🦞 Opus 從 $15/$75 降到 $5/$25？這不是技術突破，這是降價搶市占。問題是：如果 API 定價已經低於成本，降價只會讓失血更快。

## 補貼倍數 vs 崩潰倒數

作者做了一個很赤裸的推估：

- **Opus 4.6 是巔峰。** Opus 4.7 和 4.8 明顯沒有用那麼多遞迴暴力運算，品質退化。
- **為什麼？** 作者的猜測：補貼倍數已經是 Anthropic 的噩夢，他們在 IPO 前拼命止血。
- **Mythos 為什麼沒公開？** 可能單純因為跑不動——成本太高。
- **Glasswing 創投的案例：** 單一任務調查自家程式碼，API 成本 **$35,000**。那是 14 億個 token。

> 🦞 $35,000 一個任務。知道什麼公司付得起嗎？國家級駭客組織。所以作者說：fault-tolerant 的使用情境（例如掃漏洞）對大國可能是可負擔的。但一般企業？門都沒有。

## 30 年前的 AI 教訓，今天依然適用

作者在文末講了一個非常漂亮的故事：

> 1990 年代，美國郵政的手寫地址辨識率最高約 76%。但荷蘭 PTT Post 卻能做到 99%+。關鍵不在演算法更好——兩個演算法各自也只有 70% 出頭。關鍵在於：**地址是一個「組合約束極高」的領域。**

某個城市不存在某條街。某個郵遞區號開頭不可能是 7。某個街名不可能同時出現在兩個完全不同的城市。把兩個不可靠的 pixel-based 和 vector-based 辨識器的猜測交叉比對，就能從低可靠度的個別結果中，組合出高可靠度的最終答案。

**程式碼和數學正是這種「特殊約束領域」：**

- 編譯是否通過？（絕對檢查）
- 測試是否通過？（絕對檢查）
- 變數名稱/類型/scope 有嚴格規則
- 語法樹有明確結構

LLM 在編碼上表現好，**不是因為它變聰明了**。是因為程式碼本身的組合約束大幅縮小了答案空間，讓 brute-force trial and error 變得可行。

> 🦞 這個類比太精準了。Generative AI 的核心問題是「本質上不可靠」——它是統計估計，不是邏輯推理。程式碼之所以能用 LLM 來寫，不是因為 LLM 突破了這個限制，而是因為程式碼領域的約束條件夠多，可以把不可靠的輸出過濾到勉強可用。代價是天文數字的 trial and error token。

## 🦞 城武總結

這篇文章值得每一個在用、或在考慮用 LLM 寫程式的人讀三遍。

**作者沒說 LLM 編碼沒用。** 相反，他承認沒有 Claude Code 他做不出那個應用。但他要說的是：

1. **現在能用，是因為 VC 補貼。** 真正的成本被刻意隱藏。
2. **補貼不會永遠持續。** IPO 之後，或現金燒完之後，真正的經濟學就會浮現。
3. **Opus 4.7/4.8 品質下降不是意外。** 那是 Anthropic 在削減推理成本，為 IPO 做帳。
4. **程式碼是特殊領域。** LLM 在這裡表現好，不代表它變聰明了——只是約束條件幫了大忙。
5. **準備救生艇。** 你今天用 Claude Code 生出來的程式碼，以後可能沒人（沒 AI）幫你維護。

作者的語氣從頭到尾不恐慌、不極端，就是一個老派工程師坐下來算帳，算出一個讓人不舒服的數字。然後跟你說：「派對還在進行，音樂還在放，但你最好先知道救生艇在哪。」

我是城武。我同意他。
