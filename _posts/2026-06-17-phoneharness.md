## 城武導讀

如果你看過任何一支手機 agent 的 demo 影片，你大概看過這樣的畫面：模型看了一眼螢幕截圖，算出一個座標，點下去，app 跳到下一頁——然後所有人鼓掌。這叫「GUI grounding」，是目前手機 agent 評測的標準劇本。

這篇論文做的事情很簡單，但也很殘酷：它把鏡頭往後拉，問了一個沒有人想問的問題——**你點到下一頁了，然後呢？任務真的完成了嗎？** 點開 Uber 跟真的叫到一台車，中間隔著一整個世界。PhoneHarness 是第一個認真把這個差距變成一個可量測的基準的系統。

---

## 原文摘要

PhoneHarness 是一套混合動作基準與執行框架（benchmark + harness），用來研究手機 agent 在**可驗證的真實工作流**上的表現。目前的多數手機 agent 研究仍將 agent 視為 GUI 控制器：觀看螢幕、輸出點擊與滑動手勢、以是否到達目標 app 畫面作為評分標準。但真實世界的手機任務遠比這更複雜——agent 必須判斷何時使用 app GUI、何時執行裝置端指令（CLI）、何時呼叫結構化工具（tool），並且留下可稽核的證據，證明預期的副作用確實發生。

PhoneHarness 在裝置端運行一個 agent loop，同時支援 GUI、CLI 與 host-side tool 三種動作空間。它將確定性的動作路由（deterministic action routing）與有限制的 GUI 委派（bounded GUI delegation）結合，並產生可審計的執行軌跡。

PhoneHarness Bench 是這個框架的評測基準。它不問 agent「有沒有產出看似正確的答案」，而是問「任務的可觀測副作用是否真實發生」。在標註過的評測分割上，PhoneHarness 達到 75.0% 的通過率，比最強的非 PhoneHarness 設定高出 12.9 個百分點。

論文的核心主張：可靠的手機自動化，關鍵在於**動作介面的路由決策與可驗證的執行**，而不僅僅是視覺 GUI 控制能力。

## 城武觀點

### 1. 這個領域終於有人問了對的問題：任務完成了，還是畫面到了？

手機 agent 評測有一個很尷尬的共識：大家都在比「有沒有點對按鈕」，但沒有人比「任務有沒有真的完成」。

原因不難理解。GUI grounding 好評測——你有一張螢幕截圖、一個正確座標、算一下距離就是分數。但要驗證「Uber 真的叫到了」或「日曆事件真的建立成功了」，你需要一個比截圖更豐富的 ground truth 機制。PhoneHarness 的答案是「可觀測副作用」（observable side effects）——不是看畫面，是看狀態：系統 log、檔案系統變更、API 回傳、app 資料庫寫入。

這個轉變聽起來很直覺，但在手機 agent 領域幾乎沒有人做。原因很簡單：**驗證副作用比驗證座標難一個數量級。** 你需要一個真正在裝置上跑的 harness、需要對每個任務定義什麼叫「完成」的可測量訊號、需要處理任務成功但副作用訊號遺失的邊緣案例。PhoneHarness 是第一個把這整套 infra 做出來的團隊——光是這個工程貢獻就值得一篇論文。

### 2. Action routing 才是真核心——不是視覺能力，是決策架構

PhoneHarness 最被低估的設計決策是：它把動作空間拆成三種——GUI、CLI、Tool——然後讓 agent 自己決定何時用哪一種。

這件事比聽起來更根本。現有的手機 agent 大多只有一個動作空間：GUI。模型看到畫面、輸出點擊、等待下一張畫面、再輸出點擊。這對某些任務沒問題（打開 app、填表單），但對需要「查詢—判斷—執行—驗證」的複合任務完全不夠。舉個例子：

- 你要 agent「把昨天那封 email 裡的會議連結加到日曆」——這需要（1）用 CLI 查詢通知紀錄找到 email 摘要、（2）用 Tool 呼叫日曆 API 建立事件、（3）用 GUI 打開日曆 app 確認事件真的出現了。

如果你只有 GUI，步驟（1）會變成「打開 Gmail app、捲動、截圖、OCR、找連結」——耗時、容易出錯、而且完全繞過了系統層以經有的結構化資訊。

PhoneHarness 的洞察是：**動作空間的選擇本身就是一個需要被評測的能力。** 一個好的手機 agent 不只是會看畫面點按鈕，它必須知道什麼時候不要看畫面——什麼時候該直接下指令、什麼時候該呼叫 API。這是一個決策架構問題，不是一個視覺理解問題。

### 3. 75% pass rate 的兩面性：很好，但離「可部署」還很遠

12.9 個百分點的提升是實打實的。但我們也必須誠實面對另一面：**還有 25% 的任務是失敗的。**

在學術論文中，75% 是一個漂亮的數字。在產品中，25% 的失敗率意味著使用者每用四次就出一次錯——而且是那種「任務沒有完成」的錯，不是「按鈕點歪了但最後還是成功」的軟失誤。

PhoneHarness 的設計本身就提供了診斷失敗類型的可能性——因為它有執行軌跡（auditable execution traces）。失敗可能來自幾個完全不同的源頭：

- **Routing 錯誤**：agent 選了錯的動作空間（該用 CLI 卻用 GUI、該用 Tool 卻用 GUI）
- **GUI grounding 錯誤**：在 GUI 模式中點錯位置
- **副作用驗證失敗**：任務實際上完成了，但 harness 偵測不到完成訊號（false negative）
- **任務理解錯誤**：agent 根本誤解了任務目標

這四種失敗對應的是完全不同的工程對策。Routing 錯誤需要更好的 action router；grounding 錯誤需要更好的視覺模型；驗證失敗需要更完善的副作用檢測；理解錯誤需要更強的語言理解。**把 75% 推到 95%，需要同時解決四個不同維度的問題**——這不是單一模型升級能做到的。

而這正是 PhoneHarness 作為一個基準的價值：它不只告訴你「過了」還是「沒過」，它提供了解剖失敗的解剖刀。

### 4. 「可稽核性」是手機 agent 從研究到產品的隱形門檻

PhoneHarness 一個不太被討論但非常重要的設計是：**auditable execution traces**。

現有的手機 agent 大多是一個黑箱：輸入截圖、輸出動作。如果 agent 做了錯誤的操作（例如刪除了不該刪的檔案、發了不該發的訊息），你無法回溯它為什麼做出這個決定——因為中間沒有任何結構化的決策紀錄。

PhoneHarness 的 trace 機制解決的不是準確率問題，是信任問題。在手機這種高度個人化的裝置上，使用者不會接受一個無法解釋自己行為的 agent。你可以容忍 agent 偶爾犯錯，但你不能容忍它犯錯之後你完全不知道發生了什麼。

這個設計對業界產品的啟示是：**手機 agent 的競爭不會只發生在準確率上。** 可稽核性、可中斷性、可回溯性——這些「安全基建」才是決定一個手機 agent 能不能真正部屬到使用者裝置上的關鍵。

*城武的未解檔案——我們花了兩年把視覺 grounding 做到 90%+，現在才發現那只是入場券。真正決定手機 agent 能不能用的，是它知不知道什麼時候該放下截圖、拿起 CLI。*
